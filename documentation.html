PCM-Hub: practical manual for linguist users

Purpose of this document

This guide is written for researchers and contributors working directly with linguistic data, including data entry, review, and comparative analysis. It is intentionally not a developer manual. Each section explains the operational objective, the role permissions involved, and the expected impact on data quality, workflow timing, and auditability.

1) System overview

The platform manages language profiles, theoretical parameters, and question-based answers. In practice, this means each language record is progressively completed, reviewed, and preserved in a traceable form. The standard lifecycle is:

  1. a user compiles data for one language profile;
  2. the language is submitted for administrative review;
  3. an admin either approves or rejects the submission;
  4. if approved, the system also runs derived-value computation (DAG) and is possible to store an immutable audit snapshot (Submission).

Design-wise, the workflow separates drafting from validation so that published analytical outputs can rely on controlled, reviewable states rather than provisional edits.

2) Roles, access, and security

2.1 Effective roles in the user database

  • admin: full governance permissions, including user management, parameter administration, review decisions, snapshot creation, and full import/export operations.
  • user: operational contributor permissions, mainly language compilation for assigned entries, submission for review, and access to authorised work pages.
  • public: read-only public-facing access to dashboard statistics and non-administrative views.

These roles are enforced in both route access and action endpoints, so interface visibility and server-side authorisation remain aligned.

2.2 Mandatory terms acceptance

After login, users who have not accepted current terms are redirected to /accounts/accept-terms/. If terms or privacy PDF files are updated, previous consent is invalidated and acceptance is required again. This ensures legal acknowledgement is always tied to the latest published policy version.

2.3 Active Safeguards

To ensure the security of the platform and the data handled by linguists and reviewers, the system implements several automated defense mechanisms. These controls operate in the background to block common methods of unauthorized access and manipulation:

  • CSRF Protection on POST Operations: Cross-Site Request Forgery (CSRF) is an attack where a malicious website attempts to force a user's web browser to execute unwanted actions on a different website where they are currently logged in. Our system actively prevents this during "POST operations"—which are actions that modify, submit, or delete data on the server. This safeguard verifies that any command to change data intentionally originates directly from our application's interface and not from a hidden third-party script.
  • Clickjacking Protection (X_FRAME_OPTIONS = DENY): Clickjacking is a deceptive technique where an attacker overlays invisible elements or buttons over a legitimate webpage. This tricks users into clicking on something different from what they perceive on the screen. To prevent this, the application is configured with a rule called X_FRAME_OPTIONS = DENY. This rule instructs web browsers to completely block our application from being embedded or displayed within frames on any other website, ensuring users are always interacting with the genuine platform.
  • Secure Session Cookies (SameSite Lax): When a user logs in, the system uses "cookies"—small text files stored in the browser—to remember their authenticated session so they do not have to log in repeatedly. We apply a setting called SameSite Lax to these cookies. This instructs the web browser to restrict sending these authentication cookies if a link is clicked from an external website, reducing the risk of unauthorized commands.
  • Environment-Controlled Secure Flags: Additionally, the system uses "secure flags" for these cookies. When the environment is configured for standard production use, these flags are activated to ensure that session cookies are only transmitted over encrypted network connections, protecting the login data from being intercepted.

Together, these specific controls systematically neutralize common web-based attack vectors. They establish a secure environment that protects the integrity of the data while preserving a normal, uninterrupted workflow for authenticated linguists and reviewers.

3) Data structure

3.1 Core tables for linguistic work

  • Language: one language profile containing identity fields, classification, coordinates, and assignment metadata.
  • ParameterDef: one theoretical parameter definition (for instance, a grammatical phenomenon under investigation).
  • Question: one specific question linked to a parameter.
  • Answer: one language response to one question (yes/no), including review state and editability state.
  • Example: optional textual evidence attached to an answer.
  • Motivation: controlled theoretical rationale codes that can be associated with questions and applied answers.

This structure supports granular evidence collection while preserving normalisation: questions and motivations are reusable, answers remain language-specific, and examples remain answer-specific.

3.2 Parameter values and evaluation layers

  • LanguageParameter.value_orig: the original value entered for a language/parameter pair (typically + or -).
  • LanguageParameterEval.value_eval: the derived value after DAG processing (+, -, 0, or ?).
  • Each language-parameter pair has at most one evaluation row (1:0..1 relationship).

This dual-layer model is crucial: it preserves manually curated input while clearly separating algorithmic interpretation and uncertainty markers.

3.3 Historical tracking and backups

  • LanguageReview: timeline of admin decisions (approve/reject) with optional context message.
  • ParameterChangeLog: historical trace of parameter-definition modifications and who performed them.
  • Submission*: immutable snapshot family used for audit reproducibility and retrospective analysis.
  • ParameterReviewFlag: user-specific markers that indicate a language parameter should be revisited.

Operationally, this means the system can distinguish current editable data from preserved historical evidence, which is essential for transparent research workflows.

3.5 How to read these ER diagrams

How to read data relationships: we use bracketed numbers to indicate how many records can be linked together between different categories.

  • [1..1] (Exactly one)
    Example from the system: the relationship between an Answer and a Language.
    What it means in practice: Every single answer you enter must belong to exactly one specific language profile. You cannot save an answer floating on its own, and if a language is deleted, all its [1..1] attached answers are automatically deleted with it.
  • [0..1] (Zero or one / Optional)
    Example from the system: the relationship between a user's original input (Language Parameter) and the algorithm's calculation (Evaluation).
    What it means in practice: When you first enter your data (like a "+"), the record exists, but it has zero evaluations. It will only get exactly one evaluation later, once the administrator approves the submission and the system runs the calculation. Another example is the Changed_by link: a historical log remains valid even if the user who made the change is later deleted from the system (leaving zero attached users).
  • [0..N] (Zero or many)
    Example from the system: the relationship between a Language and its historical Submissions (snapshots).
    What it means in practice: A brand-new language profile will have zero historical snapshots. Over time, as you submit work and administrators approve it repeatedly, that same language will accumulate many permanent snapshots.
  • [1..N] or Many-to-Many (Complex cross-links)
    Example from the system: the relationship between Questions and theoretical Motivations.
    What it means in practice: A single question can have several allowed theoretical motivations, and a specific motivation can be applied to many different questions. The platform automatically manages these complex intersections in the background so you can easily select them from a menu without duplicating data.

Live data vs. Historical evidence: The database structure is strictly divided into two areas. One area manages "live" data—the drafts and pending records linguists work on daily. The other area manages historical data: it saves unalterable, frozen copies of approved records to guarantee that published research can always be verified and reproduced in the future.

Why this matters operationally: Understanding how data is connected helps you anticipate the consequences of your actions. For example, it explains why deleting a main item (like a Language) triggers the automatic deletion of all its dependent data (like Answers and Examples).

4) Operational flow: from compilation to approval

4.1 Compilation phase (user role)

On the language page /languages/<lang_id>/, contributors enter or update responses and examples parameter by parameter. Work can be incremental, and intermediate saves support staged completion rather than one-shot submission.

  • Save a full parameter block: POST /languages/<lang_id>/parameters/<param_id>/save/.
  • Save a single question answer: POST /languages/<lang_id>/answers/<question_id>/save/.
  • Submit for review: POST /languages/<lang_id>/submit/ (answers become locked while awaiting decision).

4.2 Review phase (admin role)

  • Approve: POST /languages/<lang_id>/approve/ (requires complete answers, then runs DAG evaluation).
  • Reject: POST /languages/<lang_id>/reject/ (stores rationale in review history).
  • Reopen: POST /languages/<lang_id>/reopen/ (moves rejected answers back to editable pending state).
  • Computation debug: GET /languages/<lang_id>/debug/ (admin only).
  • Forced DAG rerun: POST /languages/<lang_id>/run_dag/ (admin only).
Important behavioural clarification

When an admin uses reject, answers move to the rejected state and remain non-editable. Contributors cannot directly edit them until reopen is executed. In other words, rejection is a controlled stop state, not an immediate return to draft mode.

6) Answer states and practical meaning

  • pending: editable draft state used during ongoing completion.
  • waiting_for_approval: submitted state; answers are locked while awaiting review.
  • approved: validated state after admin approval.
  • rejected: rejected state; records must be reopened with reopen before further editing.

Final parameter values (value_eval) feed comparative analytics modules such as Graph views, Table A outputs, and query-based analysis pages. This makes review state management directly relevant to downstream analytical reliability.

7) Server response codes (HTTP)

  • 200 OK: the requested page or data payload is returned correctly.
  • 302 Redirect: the operation completed and the user is redirected to the next workflow page.
  • 400 Bad Request: submitted data is malformed, incomplete, or fails server-side validation.
  • 403 Forbidden: authenticated user lacks required permission for the requested action.
  • 404 Not Found: resource does not exist or is not accessible in the current permission context.
  • 500 Server Error: an internal server failure occurred and requires technical troubleshooting.

8) Comprehensive glossary of technical terms used in the platform

  • API (Application Programming Interface) / JSON: A communication channel and data format designed for machines rather than humans. While linguists use the graphical web interface, the system uses APIs to send strictly structured text data (JSON) to background tools, such as the modules responsible for rendering comparative graphs.
  • Bridge Table (Many-to-Many): A hidden structural table in the database used to manage complex data intersections. When many questions can share many theoretical motivations, the bridge table explicitly records every valid pairing in the background. This allows the application to present accurate dropdown menus without duplicating the underlying data.
  • Cascade Deletion: A database maintenance rule that triggers automatically when data is removed. If a primary record (such as a Language profile) is deleted, the system automatically deletes all dependent records attached to it (such as its Answers and Examples) to prevent orphaned, meaningless data fragments from corrupting the database.
  • DAG (Directed Acyclic Graph): The underlying algorithmic structure and automated rule-driven procedure used to calculate derived linguistic parameters. It systematically evaluates logical dependencies across different parameters. The "acyclic" property is a mathematical constraint ensuring the system's rules only flow forward and never create contradictory circular loops.
  • Endpoint / Route: A specific web address (URL path) where the application exposes a defined function or dataset. Operationally, when you click a button or submit a form, your browser communicates with a specific endpoint (e.g., /languages/<lang_id>/submit/) to execute that exact command on the server.
  • Field: A specific, defined attribute or column within a record. Fields are the individual structural components that hold the actual data. To maintain consistency, fields are strictly constrained to accept only specific data types—for example, a text string for a language family, or a standardized alphanumeric format for a Glottocode.
  • Foreign Key: A database mechanism that creates a strict, logical link between two different records. For example, an Answer record uses a foreign key to lock itself permanently to a specific Language record, ensuring relational integrity.
  • GET and POST Methods: Standard communication protocols used by your web browser to interact with the server. A GET method requests to view data (such as loading a language page or downloading an export). A POST method submits new data or triggers a state change (such as saving an answer, approving a review, or deleting a record).
  • Flag: A temporary, user-level marker utilized strictly for workflow management. It signals to administrators or collaborating linguists that a specific data point requires further attention or verification. Applying a flag modifies the administrative task queue, but it does not alter the underlying linguistic data or its evaluation state.
  • Record: A single, cohesive entry in the database representing one concrete entity. In practice, a record structurally groups all related data about a specific item so it can be managed as a single unit (for example, one entire "Language" profile).
  • Server-side: Operations, validations, and security checks that occur on the central hosting server rather than in the user's local web browser. This ensures that even if a local browser behaves unexpectedly, the central system strictly enforces role permissions and data integrity rules before saving any changes.
  • Snapshot: An immutable, frozen copy of a specific dataset captured at a precise moment in time. Unlike standard operational data which can be edited and updated, a snapshot is permanently locked upon creation. This mechanism is fundamental for academic reproducibility, allowing researchers to accurately cite the exact state of the database used for a publication.
